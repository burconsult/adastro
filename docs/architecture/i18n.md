# Internationalization

## Scope

AdAstro localizes the public site only. Admin routes remain English-only by design so editorial workflows stay deterministic.

## Runtime Model

- Public routing is locale-prefixed: `/{locale}/...`
- Active locales are controlled by:
  - `content.defaultLocale`
  - `content.locales`
- Unprefixed public requests redirect to the default locale.
- Core UI strings live in `src/lib/i18n/messages/<locale>.json`
- Feature UI strings live in `src/lib/features/<feature>/messages/<locale>.json`

## Translation Pack Contract

Every core and feature locale JSON must include the metadata keys below:

```json
{
  "_meta.locale": "nb",
  "_meta.catalogVersion": "1.1.0",
  "_meta.schemaVersion": "1",
  "_meta.fallbackLocale": "en"
}
```

Rules:

- `_meta.catalogVersion` tracks the current shipped translation catalog version.
- `_meta.schemaVersion` tracks the message-pack structure contract.
- `_meta.fallbackLocale` must currently be `en`.
- Metadata keys are stripped before runtime message resolution.

## Fallback Behavior

- Core strings fall back to English when the requested locale pack is missing.
- Feature strings fall back to English when a feature locale pack is missing or incomplete.
- A locale can only be activated when a core locale pack exists.
- Missing or version-mismatched feature packs are surfaced in `/admin/locales`.

## Admin Locales Page

`/admin/locales` is the operational control plane for public multilingual support.

It shows:

- available locale packs discovered from core and features
- active locales and the default locale
- pack health by locale
- feature fallback usage
- catalog/schema version mismatches

This page is intentionally informational plus activation-focused. It does not localize the admin.

## Adding a New Locale

1. Add `src/lib/i18n/messages/<locale>.json`
2. Add optional feature packs under `src/lib/features/*/messages/<locale>.json`
3. Keep metadata aligned with the current catalog/schema versions
4. Add translated menu labels/locale names if relevant
5. Activate the locale in `/admin/locales`
6. Add content translations for posts, pages, and navigation entries
7. Run:
   - `npm run test:run`
   - `npm run build`
   - `npm run ci:check-admin-consistency`
   - `npm run ci:check-release-hygiene`

## Best Practices

- Treat English as the canonical fallback pack.
- Do not activate a locale until the core pack is complete enough for public use.
- Keep feature packs versioned in lockstep with core pack changes.
- Prefer adding new strings to English first, then backfilling translations.
- Avoid hardcoded public UI strings outside the locale runtime.
